# Promptful API

FastAPI + SQLAlchemy (async) + PostgreSQL backend for managing LLM prompts, with full version
history kept per slug. See [/CONTEXT.md](../CONTEXT.md) for the domain vocabulary (Prompt, Slug,
Version, Live Version, Tombstone), [/docs/adr/](../docs/adr/).

## Setup

```bash
uv sync                  # install dependencies into .venv
docker compose up -d     # postgres (ply_db) + adminer (ply_adminer)
```

Connection settings are read from environment variables (see `app/core/config.py`), with an
`.env` file in `api/` as the local override. Defaults match `docker-compose.yml`:

| Variable           | Default       |
|--------------------|---------------|
| `POSTGRES_USER`     | `postgres`    |
| `POSTGRES_PASSWORD` | `ply_example` |
| `POSTGRES_HOST`     | `localhost`   |
| `POSTGRES_PORT`     | `5433`        |
| `POSTGRES_DB`       | `app`         |

Create the dev and test databases (once, on a fresh Postgres container) and apply migrations to
both:

```bash
docker exec ply_project-ply_db-1 psql -U postgres -c "CREATE DATABASE app;" -c "CREATE DATABASE app_test;"
uv run alembic upgrade head                    # dev database (POSTGRES_DB=app)
POSTGRES_DB=app_test uv run alembic upgrade head   # test database
```

## Running

```bash
uv run fastapi dev app/main.py     # dev server with reload, http://localhost:8000
uv run uvicorn app.main:app        # equivalent, without the fastapi CLI wrapper
```

Interactive API docs are served at `/docs` (Swagger UI) and `/redoc`.

## Testing

```bash
uv run pytest                               # full suite
uv run pytest tests/test_prompt_schemas.py  # unit tests only ŌĆö no database needed
uv run pytest tests/test_prompt_routes.py   # integration tests ŌĆö needs the test database, below
uv run pytest tests/test_prompt_routes.py::test_create_returns_201_with_version_1  # single test
```

- `tests/test_prompt_schemas.py` ŌĆö unit tests for the Pydantic-layer validation (slug format,
  Jinja2 syntax checking) with no database involved.
- `tests/test_prompt_routes.py` ŌĆö integration tests that drive the real FastAPI app over HTTP
  (via `httpx.AsyncClient` + `ASGITransport`) against the `app_test` Postgres database.
  Migrations must already be applied to `app_test` (see Setup above) before running these.
  Each test's writes are wiped by a `TRUNCATE TABLE prompts` that runs automatically after the
  test (see `tests/conftest.py`).

## Migrations

Alembic runs over the sync `psycopg2` driver, separate from the app's async `asyncpg` runtime
engine ŌĆö both are already in `pyproject.toml`'s dependencies for this reason.

```bash
uv run alembic revision --autogenerate -m "describe the change"   # after editing app/models/
uv run alembic upgrade head
```

Review autogenerated migrations before applying ŌĆö they pick up table/column/constraint
additions, but always double-check anything involving raw SQL (e.g. `CheckConstraint` text) since
autogenerate's detection of those can be inconsistent across SQLAlchemy/Alembic versions.

## API

Base path: `/prompt`. Full contract (request/response shapes, status codes) is documented in
[/product/ai-specs/add-prompt-endpoint.md](../product/ai-specs/add-prompt-endpoint.md#api); the
short version:

| Method   | Path             | Purpose                                              |
|----------|------------------|-------------------------------------------------------|
| `POST`   | `/prompt/create` | Create a Prompt at a new (or currently-deleted) slug |
| `POST`   | `/prompt/{id}`   | Update ŌĆö append a new Version on top of `{id}`        |
| `DELETE` | `/prompt/{id}`   | Delete ŌĆö append a Tombstone Version on top of `{id}`  |
| `GET`    | `/prompt/{id}`   | Fetch one exact Version by its row id                 |
| `GET`    | `/prompt`        | Fetch by `?slug=`, optionally pinned to `&version=`   |

There is no `PUT` and no row is ever updated or physically deleted in place ŌĆö see
[ADR-0001](../docs/adr/0001-versions-are-immutable-rows.md) and
[ADR-0003](../docs/adr/0003-no-put-optimistic-concurrency-via-id.md) for why.
